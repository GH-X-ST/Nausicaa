Nausicaa Real-Flight Experiment Instructions
============================================

Purpose
-------
This folder contains the self-contained real-flight runtime. The live loop is:

Vicon rigid body Nausicaa -> canonical 15-state -> balanced-cluster frozen governor
-> 20 percent command lattice -> Nano33 IoT packet -> transmitter/receiver/servos.

The real-flight experiment launcher lets the operator start one case once. Before
each throw it holds neutral for a short pre-arm delay with Vicon inactive, then
starts Vicon launch-gate tracking, records one gate-passed throw, sends neutral
after exit, waits for cooldown, and automatically re-arms for the next throw.
Failed launch starts are ignored: they do not count as valid throws and do not
update memory.


Before Any Throw
----------------
1. Power the receiver, transmitter path, Nano33 IoT, and Vicon system.
2. In Vicon, confirm the glider rigid body is named exactly:
   Nausicaa
3. If fan tracking is used, confirm fan rigid bodies are named exactly:
   Fan_1, Fan_2, Fan_3, Fan_4
4. Confirm the Vicon origin and yaw alignment match the runtime settings.
   The runtime applies the calibrated full x/y/z offset to map raw Vicon
   position into the controller frame.
5. Install pyserial in the active Python environment if needed:
   C:\ProgramData\miniforge3\python.exe -m pip install pyserial
6. Run the surface sign check before closed-loop flight.


Vicon Frame Calibration
-----------------------
Before throwing, calibrate the Vicon-to-arena offset. Hold the glider still at a
known physical arena coordinate, edit KNOWN_ARENA_POINT_M at the top of:

04_Flight_Test\01_Runtime\run_vicon_frame_calibration.py

Then run:

C:\ProgramData\miniforge3\python.exe 04_Flight_Test\01_Runtime\run_vicon_frame_calibration.py

The script prints the raw Vicon mean, current transformed mean, current error,
and recommended VICON_POSITION_OFFSET_M. Paste the recommended offset into
run_experiment_sequence.py if the axis directions are correct:

VICON_POSITION_OFFSET_M = (...)

If moving the glider in physical +x, +y, or +z causes the wrong transformed
coordinate to change, the axis mapping must be corrected before flight.


Surface Sign Check
------------------
Run:

C:\ProgramData\miniforge3\python.exe 04_Flight_Test\01_Runtime\run_surface_sign_check.py --mode armed --serial-port COM11

The script sweeps aileron, elevator, and rudder through the full 20 percent
command lattice [-1.0, -0.8, ..., +0.8, +1.0] while printing live progress.
Watch each printed command and confirm the physical surface motion:
1. neutral: all surfaces neutral.
2. positive aileron values: left and right ailerons move in opposition for positive roll.
3. negative aileron values: opposite of positive aileron.
4. positive elevator values: elevator moves for nose-up positive pitch.
5. negative elevator values: opposite of positive elevator.
6. positive rudder values: rudder moves for nose-right positive yaw.
7. negative rudder values: opposite of positive rudder.
8. neutral_end: all surfaces neutral.

If any surface moves the wrong way, stop. Do not fly until servo sign, receiver
channel order, and packet mapping are fixed.


Vicon Orientation Check
-----------------------
Before any closed-loop throw, confirm that the Vicon rigid body orientation
matches the controller body-frame convention:

C:\ProgramData\miniforge3\python.exe 04_Flight_Test\01_Runtime\run_vicon_orientation_check.py

The script guides neutral hold, +x/+y/+z translations, pitch up/down, roll
right/left, and yaw right/left. Each step starts after a 10 s countdown, so the
operator can reposition the glider without pressing Enter. It writes samples and
a pass/fail summary under:

04_Flight_Test\05_Results\vicon_orientation_check\<run_label>

Do not fly if any sign check fails. The most critical crash-prevention check is:

nose-up physical pitch -> positive theta.

For the pitch, roll, and yaw steps, start near the neutral/aligned pose during
the countdown, then move smoothly during the collection window. The checker now
also verifies the SO(3) body-rate observer:

nose-up motion -> positive q
right-wing-down roll motion -> positive p
nose-right yaw motion -> positive r
rate confidence >= 0.65
measured Vicon sample rate close to the requested 200 Hz

The current runtime applies the recovered attitude correction
`VICON_ATTITUDE_SIGNS = (1, -1, -1)`, so roll is unchanged while pitch and yaw
are flipped before the controller sees the state. Re-run this checker after the
fix and only continue if pitch-up and yaw-right now pass.


How To Start One Experiment Case
--------------------------------
Open:

04_Flight_Test\01_Runtime\run_experiment_sequence.py

At the top of the file, edit:

CURRENT_EXPERIMENT_CASE = "E0.1"
CURRENT_SESSION_LABEL = ""
SERIAL_PORT = "COM11"
VICON_HOST = "192.168.0.100:801"
MODE = "armed"

Then run:

C:\ProgramData\miniforge3\python.exe 04_Flight_Test\01_Runtime\run_experiment_sequence.py

The current E0.1 shakedown pre-arm delay is 3 s. During this delay and during the 5 s
between-throw cooldown, Vicon is inactive and the Nano receives neutral commands
only. After the pre-arm delay, the console prints that Vicon launch-gate
tracking is active; this is the time to throw through the start gate.
The same 5 s cooldown is used after a failed launch attempt. A launch-plane
crossing faster than 2 m/s that fails the formal launch gate is marked as an
invalid attempt immediately, stored under `invalid_attempts`, excluded from
valid throw count and memory, and then re-armed after the cooldown.
During launch wait and active flight, the runtime polls Vicon at 200 Hz while
using an 8 Hz velocity filter and a corrected SO(3) body-rate observer, keeping
the governor at 10 Hz and the serial command repeat at 50 Hz.

The console prints the case ID, controller mode, memory mode, target valid
throw count, latest completed throw, launch speed, termination reason,
controller decision count, max decision time, fan count, and memory cell count.


Launch Gate Rule
----------------
A throw becomes active at the first measured state that satisfies the complete
R5 launch window:

x_w in [1.2, 1.4] m
y_w in [1.8, 2.2] m
z_w in [1.3, 1.8] m
roll in +/-20 deg
pitch in [-10, +20] deg
yaw in +/-20 deg
speed in [3, 8] m/s
p roll rate in [-1.2, +1.2] rad/s
q pitch rate in [-1.2, +1.2] rad/s
r yaw rate in [-1.8, +1.8] rad/s
SO(3) body-rate observer confidence >= 0.65
2 consecutive approved frames

The runtime also computes the interpolated x_w = 1.3 m launch-plane crossing as
a fallback and diagnostic, but the intended real-flight trigger is the first
full-window-valid sample with sufficient angular-rate confidence. The rate
observer stays warm during prelaunch neutral tracking; it is not reset at the
start of active control. This prevents one noisy Vicon frame from starting the
controller.

If the launch gate times out, the attempt is stored under invalid_attempts, but
it does not count as a throw and does not update memory.


Fan Layout Rule
---------------
Never move fan positions during one flight.
Never move fan positions inside one memory episode.
Move fans only between separate layout episodes.

Fan markers are logged for evidence and sanity checking. They do not create
fan-layout-specific controller branches.


Experiment Checklist
--------------------
E0.1 - Dry-air shakedown, 5 valid throws.
  Fan markers: none expected.
  Purpose: confirm launch gate, Vicon transform, neutral recovery, and logging.

E0.2 - Four-fan shakedown, 5 valid throws.
  Fan markers: Fan_1 to Fan_4 expected.
  Purpose: confirm fan tracking and closed-loop logging with fans active.

Case suffix rule:
  .0 = open-loop neutral baseline.
  .1 = closed-loop controller, no memory.
  .2 = closed-loop controller, memory enabled.

E1.0 - Dry air, open-loop neutral baseline, 10 valid throws.
E1.1 - Dry air, no memory.
E1.2 - Dry air, memory enabled.
  Fan markers: none expected.
  Purpose: dry-air baseline and launch-speed dependence.

E2.0 - Single fan fixed, open-loop neutral baseline, 10 valid throws.
E2.1 - Single fan fixed, no memory.
E2.2 - Single fan fixed, memory enabled.
  Fan markers: one fan expected.
  Purpose: clearest real updraft memory-learning evidence.

E3.0 - Four fan fixed, open-loop neutral baseline, 10 valid throws.
E3.1 - Four fan fixed, no memory.
E3.2 - Four fan fixed, memory enabled.
  Fan markers: four fans expected.
  Purpose: main multi-updraft exploitation evidence.

E4a.0/E4a.1/E4a.2 - Hard shifted layout 1, open-loop / no memory / memory.
E4b.0/E4b.1/E4b.2 - Hard shifted layout 2, open-loop / no memory / memory.
E4c.0/E4c.1/E4c.2 - Hard shifted layout 3, open-loop / no memory / memory.
  Fan markers: four fans expected.
  Purpose: unseen but fixed difficult layouts.

E5a.0/E5a.1/E5a.2 - Random layout 1, open-loop / no memory / memory.
E5b.0/E5b.1/E5b.2 - Random layout 2, open-loop / no memory / memory.
E5c.0/E5c.1/E5c.2 - Random layout 3, open-loop / no memory / memory.
E5d.0/E5d.1/E5d.2 - Random layout 4, open-loop / no memory / memory.
  Fan markers: one to four fans expected.
  Purpose: broader unseen-layout generalisation.

Open-loop neutral cases use the same launch gate, Vicon logging, fan logging,
serial path, exit gate, and result folder layout as controlled cases. After
launch approval, they send only neutral commands, make zero governor decisions,
and do not update memory.


Result Folders
--------------
Each case is stored separately:

04_Flight_Test\05_Results\<case_id>\<session_label>\throw_001
04_Flight_Test\05_Results\<case_id>\<session_label>\throw_002
...
04_Flight_Test\05_Results\<case_id>\<session_label>\invalid_attempts\attempt_001

Use the case-level experiment_sequence_summary.csv for a quick run overview.
Use each throw folder for detailed state, fan, command, controller, and memory
logs.
