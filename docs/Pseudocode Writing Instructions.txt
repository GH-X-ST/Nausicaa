## Active Transition-Aware Thesis Workflow

The active thesis workflow is `R5 -> R7 -> R8 -> R10 -> R11 -> Reality`. R9 remains internal preflight only and is not thesis-facing evidence. R10 tunes the viability governor with residual updraft adaptation, and R11 is the held-out validation gate.

Launch is an entry regime, not a separate controller family. The active primitive catalogue has exactly eight manoeuvre families: `glide`, `recovery`, `lift_entry`, `lift_dwell_arc`, `mild_turn_left`, `mild_turn_right`, `energy_retaining_bank`, and `safe_exit_or_recovery_handoff`. Retired `launch_capture_*` IDs are archive aliases only and must not appear in active evidence.

Every primitive is treated as a transition object. R5 is robust transition-aware primitive / transition-object Q/R plus primitive attitude/bank reference-bias training across five entry start families with exact dense proportions per primitive/candidate/environment: 40 `launch_gate`, 25 `inflight_nominal`, 15 `inflight_lift_region`, 10 `inflight_boundary_near`, and 10 `inflight_recovery_edge`. The dense evaluation target remains `8 * 32 * 3 * 100 = 76,800` rows, but row count alone is not a pass condition. Candidate 0 is nominal, candidates 1-7 are named physical anchors with small interpretable attitude/bank reference biases, and candidates 8-31 are deterministic Latin-hypercube log multipliers over the seven grouped LQR weights plus bounded pitch and bank reference biases. Speed-bin selection is for local model scheduling only; active primitives must not chase speed as a hard reference. R5 writes `r5_transition_candidate_training_summary.csv`, `r5_transition_selected_for_r7.csv`, `r5_transition_pareto_front.csv`, and `r5_transition_training_manifest.json`, then freezes only selected transition objects for R7 while keeping the full candidate bundle as audit evidence.

R7 is held-out transition validation of the frozen R5-selected transition objects. No primitive may pass R7 solely on local rollout success; no primitive may pass R5 or R7 from dense row count or aggregate primitive success across entry classes. A controller can survive for one `primitive_id + entry_class` and fail for another. R8 compresses transition objects grouped by `primitive_id` and `transition_entry_class` using coverage-aware medoid selection without averaging Q/R, K, references, or controller IDs.

The governor classifies the current state, filters representatives by validated `transition_entry_class`, rejects high hard-failure risk, scores transition probability plus updraft gain plus flight time plus residual-memory correction, executes the best transition object, and updates case-local residual memory. Step 0 has `current_state_class = launch_gate`, so it selects only transition objects validated for `entry_class = launch_gate`; there is no launch-specific primitive family route.

The inner LQR remains a stabilising tracker around the primitive-defined local reference. It must not become the manoeuvre planner: primitive transition objects define manoeuvre intent and R5/R7/R8 decide which transition objects are valid, while the governor selects among those frozen transition objects. Signed turn intent is diagnostic only: R5/R7 may record signed bank, signed roll-rate, and lateral turn tendency for audit, but active selection must not reward turn-expression strength and `mild_turn_left` / `mild_turn_right` must not receive sign-constrained bank or roll-rate reference forcing. Same-start trajectory plots are diagnostic-only sanity checks for controller behaviour and do not replace R5/R7 transition evidence.

Residual memory is a small case-local modifier: `predicted_updraft_gain = library_prediction + residual_memory_correction`. It must not override state classification or entry/exit compatibility. Memory is reinitialised per final test row. Final scoring is computed only from the final held-out rollout path. There is no hidden speed gate, no energy-loss hard failure, no PD/PID, no TVLQR, and no fan-layout-specific controller logic.

Core comparison is no memory versus residual-memory histories such as h5, h20, and h100, with safe-explore only as optional ablation. Hardware readiness, real-flight transfer, mission success, autonomy, and memory-improvement claims require R11 and later real-flight evidence.

## Purpose

Convert implemented code or planned methods into compact, journal-quality pseudocode. The goal is to expose the algorithmic idea, not to translate source code line by line.

Use **primitive**, not skill.

## Recommended algorithms

The main thesis or paper algorithms should be:

1. Environment-context encoding.
2. Launch-capture primitive-library generation and first-decision audit.
2. W0/W1 rich primitive-library generation.
3. W2/W3 fixed-LQR survival filtering.
4. Post-W3 library-size cross-study and representative selection.
5. Primitive outcome model construction.
6. Viability-governed primitive selection.
7. Repeated-launch belief update.
8. Sim-real replay pairing.

When controller design is shown, express it as primitive-local LQR variant learning inside W0/W1 dense generation: each primitive variant includes entry set, reference, finite horizon, local Q/R/K, exit checks, and failure labels. W2/W3 algorithms should replay fixed variants and filter by survival evidence without retuning. Post-W3 library-size cross-study should be shown only after W3. Do not write an external controller-bank algorithm.

Do not write algorithm blocks for plotting, file export, command-line parsing, folder management, or cosmetic report generation.

## Robotics-journal target

The algorithm should help a robotics reader reproduce the method. Include physical inputs, local flow context, safety checks, and outputs used in experiments.

Good:

```text
Reject primitive if the predicted trajectory leaves the safety volume or violates the recoverable-speed bound.
```

Poor:

```text
Apply filtering condition.
```

## Algorithm structure

Use:

```latex
\begin{algorithm}[t]
\small
\caption{Descriptive algorithm name}
\label{alg:descriptive_label}
\KwIn{State, flow context, primitive set, bounds, belief, latency case.}
\KwOut{Selected primitive, rejection log, episode result, or updated belief.}
...
\end{algorithm}
```

Prefer `algorithm2e` for thesis and journal-style pseudocode.

## What to include

Include:

- scientific inputs;
- measured or simulated state;
- environment-context features;
- primitive catalogue;
- admissible set or safety volume;
- primitive rollout or outcome estimate;
- acceptance/rejection condition;
- scoring function;
- output record;
- termination condition.

Do not include:

- file names;
- Python helper names;
- CLI flags;
- cache logic;
- plotting calls;
- verbose manifest fields;
- internal run IDs, unless the algorithm is specifically about a manifest system.

## Example high-level primitive selection pseudocode

```text
Input: state x, context c, primitive catalogue P, belief b, safety bounds S
For each primitive p in P:
    estimate outcome y_p from (x, c, p, b)
    if y_p violates S:
        add rejection reason
    else:
        compute utility from dwell, energy, margin, and uncertainty
Select highest-utility viable primitive, or reject all
Return selected primitive and rejection log
```

## Repeated-launch belief update idea

```text
For each episode:
    measure initial state
    compute local context
    select primitive through governor
    execute or simulate primitive
    compute lift evidence, energy residual, dwell, and termination cause
    update belief with decay
    store episode record
```

Core equation:

```text
b_{k+1}(r) = lambda b_k(r) + (1 - lambda) w_hat_k(r)
```

## Reviewer check

Before finalising pseudocode, check:

- Does it describe the method rather than implementation details?
- Are all inputs and outputs scientifically meaningful?
- Are safety and rejection conditions explicit?
- Is it short enough for a reader to follow?
- Does it connect directly to the experiment?
