# CODEX Instruction — Python Implementation for Nausicaa

## Role

You are editing Python code for a research-grade robotics/aerospace simulation and experiment pipeline. Your task is to implement only the requested change while preserving physical conventions, rich comments, runtime efficiency, and auditability.

## Before editing

1. Identify the exact files allowed to change.
2. Inspect current public interfaces, imports, state order, command order, and units.
3. Identify which outputs become evidence: manifest, CSV, plot, report, or log.
4. Do not edit unrelated code.
5. Do not reset or overwrite existing result directories unless explicitly authorised.

## Coding style

Use rich comments. Comments should explain physical assumptions, evidence boundaries, unit/frame conventions, and why a gate or metric exists.

Do not remove useful existing comments unless they are wrong.

Use:

- type hints;
- dataclasses where structured records are needed;
- explicit unit suffixes;
- deterministic seeds;
- `Path` for file paths;
- narrow helpers;
- explicit validation at module boundaries.

Avoid:

- speculative frameworks;
- generic configuration systems;
- hidden unit conversion;
- silent frame conversion;
- broad try/except that hides failures;
- new dependencies unless justified.

## Project-specific implementation priorities

For current mission work, prefer:

- fixed-gate launch-state sampling;
- repeated-launch episode records;
- decaying lift-belief update;
- primitive selection through viability governor;
- branch-local archive decisions;
- fixed-gate and local-shell envelope maps;
- real-flight replay pairing;
- compact metrics and manifests.

Do not make high-angle agile turns a required final mission dependency. Keep them as optional boundary/refinement evidence unless the project plan explicitly changes.

## Evidence and claims

Every output should state whether it is:

```text
simulation-only
hardware shakedown
real-flight evidence
partial transfer
negative transfer
instrumentation-limited
```

Do not claim real-flight transfer, mission success, or hardware readiness unless the result file directly supports it.

## Validation

Run the narrowest meaningful checks:

- import check;
- smoke run;
- schema check;
- unit test;
- small deterministic sample;
- row-count and manifest check;
- no-overwrite check.

If a check cannot run, report why. Do not imply that it passed.

## Final response

Report:

1. Files changed.
2. Interfaces added or preserved.
3. Tests/checks run.
4. Evidence files generated.
5. Claim boundaries preserved.
6. Any remaining risks.
