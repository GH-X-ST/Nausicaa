## Active Transition-Aware Thesis Workflow

The active thesis workflow is `R5 -> R7 -> R8 -> R10 -> R11 -> Reality`. R9 remains internal preflight only and is not thesis-facing evidence. R10 tunes the viability governor with residual updraft adaptation, and R11 is the held-out validation gate.

Launch is an entry regime, not a separate controller family. The active primitive catalogue has exactly eight manoeuvre families: `glide`, `recovery`, `lift_entry`, `lift_dwell_arc`, `mild_turn_left`, `mild_turn_right`, `energy_retaining_bank`, and `safe_exit_or_recovery_handoff`. Retired `launch_capture_*` IDs are archive aliases only and must not appear in active evidence.

Every primitive is treated as a transition object. R5 is robust transition-aware primitive / transition-object Q/R plus primitive attitude/bank reference-bias training across five entry start families with exact dense proportions per primitive/candidate/environment: 40 `launch_gate`, 25 `inflight_nominal`, 15 `inflight_lift_region`, 10 `inflight_boundary_near`, and 10 `inflight_recovery_edge`. The dense evaluation target remains `8 * 32 * 3 * 100 = 76,800` rows, but row count alone is not a pass condition. Candidate 0 is nominal, candidates 1-7 are named physical anchors with small interpretable attitude/bank reference biases, and candidates 8-31 are deterministic Latin-hypercube log multipliers over the seven grouped LQR weights plus bounded pitch and bank reference biases. Speed-bin selection is for local model scheduling only; active primitives must not chase speed as a hard reference. R5 writes `r5_transition_candidate_training_summary.csv`, `r5_transition_selected_for_r7.csv`, `r5_transition_pareto_front.csv`, and `r5_transition_training_manifest.json`, then freezes only selected transition objects for R7 while keeping the full candidate bundle as audit evidence.

R7 is held-out transition validation of the frozen R5-selected transition objects. No primitive may pass R7 solely on local rollout success; no primitive may pass R5 or R7 from dense row count or aggregate primitive success across entry classes. A controller can survive for one `primitive_id + entry_class` and fail for another. R8 compresses transition objects grouped by `primitive_id` and `transition_entry_class` using coverage-aware medoid selection without averaging Q/R, K, references, or controller IDs. R8 must also preserve distinct W3-surviving local LQR speed-bin coverage within each primitive/entry-class group up to the case representative budget; speed-bin collapse is a library-coverage failure, not an LQR-principle failure.

The governor classifies the current state, filters representatives by validated `transition_entry_class`, rejects high hard-failure risk, scores transition probability plus updraft gain plus flight time plus residual-memory correction, executes the best transition object, and updates case-local residual memory. Step 0 has `current_state_class = launch_gate`, so it selects only transition objects validated for `entry_class = launch_gate`; there is no launch-specific primitive family route.

The inner LQR remains a stabilising tracker around the primitive-defined local reference. It must not become the manoeuvre planner: primitive transition objects define manoeuvre intent and R5/R7/R8 decide which transition objects are valid, while the governor selects among those frozen transition objects. Same-start trajectory plots are diagnostic-only sanity checks for controller behaviour and do not replace R5/R7 transition evidence.

Residual memory is a small case-local modifier: `predicted_updraft_gain = library_prediction + residual_memory_correction`. It must not override state classification or entry/exit compatibility. Memory is reinitialised per final test row. Final scoring is computed only from the final held-out rollout path. There is no hidden speed gate, no energy-loss hard failure, no PD/PID, no TVLQR, and no fan-layout-specific controller logic.

Core comparison is no memory versus residual-memory histories such as h5, h20, and h100, with safe-explore only as optional ablation. Hardware readiness, real-flight transfer, mission success, autonomy, and memory-improvement claims require R11 and later real-flight evidence.

## Purpose

Use this guide for Python code in modelling, simulation, primitive rollout, archive execution, context-feature extraction, outcome modelling, belief updates, plotting, and real-flight analysis.

The code should be readable by a future researcher without needing a verbal explanation.

## Current method language

Use **primitive**, not skill.

The current method is transition-aware primitive selection:

```text
state + local flow context + primitive parameters
    -> primitive outcome estimate
    -> transition viability governor
    -> selected 0.10 s primitive
    -> entry_class -> exit_class transition log
    -> episode outcome
    -> directional 3D residual updraft-gain update
```

Do not encode fan layouts as separate online algorithms. Treat them as environment instances represented through context features.

Use the strict surrogate ladder:

```text
W0 is dry air only
Active pass-gated W1 evidence is annular-GP randomized training only; Gaussian plume is diagnostic-only
W2 is GP-corrected annular-Gaussian only
W3 is randomised GP-corrected annular-Gaussian only
```

W labels are fidelity/evidence layers, not online branches. Single-fan, four-fan, fan-position, active-fan-count, and fan-power cases are environment instances, not separate algorithms. In R5, dry-air rows keep the W0 no-updraft environment but still use W3-style plant and implementation perturbations; the absence of updraft must not imply a nominal-only glider or latency/actuator case. Plant perturbations must match the active `plant_instance.py` implementation: mass scale, CG-offset strip moment-arm shift, diagonal inertia scaling, strip `cd0` scaling, and flap-scale surface calibration, with cross-inertia logged as not perturbed.

## Primitive-local LQR variant learning

The LQR stabiliser is part of the primitive, not an external controller bank. Represent the active library item as a primitive-controller transition object with `primitive_variant_id`, `primitive_id`, `transition_entry_class`, `controller_id`, reference state/command, finite horizon, Q/R JSON, gain checksum, exit checks, metrics, failure labels, and claim status.

W0/W1 dense generation develops a rich primitive-controller variant library and tunes the primitive-local LQR inside every generated variant. Preserve all W0/W1 variants and evidence rows; do not reduce them to a 12--24 retained shortlist. W2 is diagnostic-only. R7/W3 replays fixed variants to eliminate or downgrade failures without regenerating or retuning LQR controllers, and no primitive may pass R7 solely from local rollout success. Clustering and merging occur only after R7. The mixed primitive-start distribution includes launch and in-flight starts; do not reject an in-flight-only primitive solely because it cannot start from the broad launch gate. Late validation uses frozen post-W3 variants and must not hide Q/R or gain retuning.

Active LQR synthesis uses local-speed scheduled operating-point linearisation. Do not reintroduce optimizer-CSV trim seeds, bare global `linearise_trim()` calls, or forced convergence to a single 6.5 m/s trim speed. The code schedules over speed bins from 2.0 to 9.0 m/s; operating points below the feasible straight-trim seed range are local operating-point linearisations, not claims of steady trim.

The Q/R and reference-bias table is deterministic candidate generation, not an
online optimiser and not a global optimality proof.  Code and reports should
call the chosen controllers R5-selected or transition-validated LQR variants
after the evidence gates pass.


## Core standard

Code must be:

- correct in frames, signs, and units;
- efficient enough for archive-scale simulation;
- rich in meaningful comments;
- traceable through manifests, metrics, and tests;
- narrow in scope.

## Rich-comment rule

Good comments explain why a line exists or what physical assumption it encodes.

Use comments for:

- units and coordinate frames;
- sign conventions;
- state and command order;
- context-feature definitions;
- controller timing assumptions, including W0/W1 feedback latency, command timing, and actuator lag;
- latency case meaning;
- safety gate meaning;
- belief update meaning;
- archive row classification;
- why a failure label is assigned;
- why a result is simulation-only or hardware-eligible.

Avoid comments that restate syntax.

## Naming and units

Use suffixes:

```text
_m, _s, _rad, _deg, _m_s, _rad_s, _kg, _N, _Nm
```

Use meaningful fields:

```text
environment_id
context_feature_vector
primitive_parameters
belief_before
belief_after
minimum_wall_margin_m
energy_residual_m
lift_dwell_time_s
termination_cause
claim_status
```

Avoid ambiguous names such as `data`, `val`, `result2`, `temp`, `out`, or `status_flag` unless the local context is trivial.

## Runtime and storage

Dense runs must use:

- chunked execution;
- resumable chunk manifests;
- compressed table partitions;
- CPU worker count chosen by profiling and memory guardrails;
- checksums and table manifests;
- no full-memory final rollout table.

Every generated file should stay below 100 MB. Prefer below 75 MB. If a table or figure may exceed 100 MB, split or simplify before writing.

## Environment context code

An environment-context encoder should expose local physical features instead of named environment branches.

Minimum useful outputs:

```text
w_cg_m_s
w_wing_mean_m_s
delta_w_lr_m_s
spanwise_w_gradient_m_s_per_m
w_local_uncertainty_m_s
lift_score
wall_margin_m
all_wall_margin_m
front_wall_margin_m
left_wall_margin_m
right_wall_margin_m
rear_wall_margin_m
governor_wall_margin_m
floor_margin_m
ceiling_margin_m
attitude_margin_rad
```

The encoder should not contain separate online logic for particular fan layouts.
For governor preselection, use `governor_wall_margin_m = min(front, left,
right)` when heading-aware margins are available; keep `wall_margin_m` as the
all-wall hard-boundary audit value. Rollout boundary checks must still enforce
all walls, floor, and ceiling. The active governor wall guard is `0.001 m`
(0.10 cm); tighter wall-margin logic must be treated as a safety-policy choice,
not a physical failure.
Do not include `speed_margin_m_s` in active context features, governor
rejection logic, recovery routing, launch scoring, or validation audit gates.
Do not block or terminate a primitive solely because speed drops below a nominal
reference; the physical rollout must decide whether the state actually causes
floor, ceiling, wall, nonfinite, or unrecoverable terminal failure.
Likewise, negative net energy or gross energy loss must never produce a hard
failure label, governor rejection, or termination gate. R9/R10 scoring should
report whole-episode net energy drift and gross energy loss as audit fields, but
score useful updraft extraction separately through an updraft-gain proxy. The
active model-backed proxy is trajectory-integrated positive wing-panel vertical
wind from the actually simulated primitive rollout, archived as
`trajectory_integrated_updraft_gain_m`,
`trajectory_mean_positive_w_wing_m_s`, `trajectory_lift_dwell_time_s`, and
`updraft_integration_status`. Primitive-start local `w_wing` exposure is only a
legacy/smoke fallback; dry/no-local-lift contexts must not inherit positive
updraft score from aggregate representative evidence, and positive
specific-energy increments are only a fallback when local updraft context is
unavailable. Governor soft scoring must use context-conditioned
`expected_updraft_gain_proxy_m`; keep `expected_energy_residual_m` only as a
legacy audit input/output alias for net specific-energy drift. Directional
memory residuals must be named `updraft_gain_residual_m`, not total-energy
residuals. Use
`updraft_gain_weight` and `terminal_updraft_gain_weight` for active governor
configuration output; accept legacy `energy_weight` names only when reading old
frozen configs. Missing compact-library outcome evidence must be rejected as
`missing_outcome_evidence_for_candidate`, not collapsed into a zero-probability
rejection.
R10/R11 must not feed raw representative-average outcome rows directly into the
governor. Use the robust context-conditioned outcome adapter first. The adapter
is downgrade-only: it can reduce continuation or terminal probability, cap
updraft/dwell soft rewards, or increase hard-failure risk based on current
start regime, environment class, local lift, local uncertainty, margins,
environment block, active fan count, and numeric positive `sample_count`
coverage. Full R10/R11 must block if the outcome table lacks usable
`sample_count` coverage. It must never make a
candidate look safer or more performant than the base post-W3 representative
evidence.
`safe_success` in R10/R11 requires a role-compliant launch -> in-flight/recovery
sequence ending in continuation-valid, terminal-useful, or explicit
time-budget completion, with no physical hard failure, floor/ceiling violation,
or no-viable primitive tail. Terminal-useful safe exits must be reported
separately from `full_safe_success`; strict R11 must gate on
`full_safe_success` and consume the frozen governor configuration written by
R10. The empty-prior baseline is named
`empty_frozen_prior_baseline`; it must not seed hand-made synthetic
observations unless a real static map is explicitly supplied under a separate
diagnostic label.

## Primitive outcome code

Outcome datasets should preserve W0/W1 rich evidence, W2/W3 survival evidence, and post-W3 library-size cross-study labels. They should record:

```text
state
context features
primitive family
primitive parameters
accepted / weak / failed / rejected / blocked
boundary_terminal for retained x/y boundary exits
continuation_status
episode_terminal_status
episode_utility_label
energy residual
dwell time
minimum margin
termination cause
failure label
```

Do not promote diagnostic, open-loop, schema-only, command-template, or blocked rows into mission evidence. Keep continuation-valid labels separate from episode-terminal-useful labels. X/y boundary exits may be useful terminal episode evidence, but they are not downstream-continuation successes. Do not run clustering or small-library selection on W0/W1 outputs; compression is a post-W3 operation.

## Validation and tests

Every evidence-generating script should have at least one of:

- smoke test;
- schema test;
- row-count check;
- manifest consistency check;
- file-size audit;
- seed reproducibility check;
- physics/sign/unit audit.

A run is not evidence unless it produces traceable outputs.

## Final response standard for coding tasks

Report:

1. files changed;
2. interfaces added or changed;
3. tests/checks run;
4. evidence files generated;
5. claim boundaries preserved;
6. remaining risks.
