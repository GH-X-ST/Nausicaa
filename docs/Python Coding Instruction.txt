# Python Coding Instruction — Environment-Conditioned Primitive Standard

Use the explicit library-size case labels `heavy_cluster`, `balanced_cluster`, `light_cluster`, and `no-clustering/no-merging` in reports or manifests where the human-readable case name is needed.

## Purpose

Use this guide for Python code in modelling, simulation, primitive rollout, archive execution, context-feature extraction, outcome modelling, belief updates, plotting, and real-flight analysis.

The code should be readable by a future researcher without needing a verbal explanation.

## Current method language

Use **primitive**, not skill.

The current method is environment-conditioned primitive selection:

```text
state + local flow context + primitive parameters
    -> primitive outcome estimate
    -> viability governor with safe exploration/exploitation
    -> selected 0.10 s primitive
    -> episode outcome
    -> directional 3D residual lift-belief update
```
Repeated-launch validation must report memory histories after 0, 5, 10, 20, 50, and 100 prior launches. These history lengths are required comparison groups, not optional diagnostics.


Do not encode fan layouts as separate online algorithms. Treat them as environment instances represented through context features.

Use the strict surrogate ladder:

```text
W0 is dry air only
W1 is variable Gaussian plume only
W2 is GP-corrected annular-Gaussian only
W3 is randomised GP-corrected annular-Gaussian only
```

W labels are fidelity/evidence layers, not online branches. Single-fan, four-fan, fan-shift, and fan-power cases are environment instances, not separate algorithms.

## Primitive-local LQR variant learning

The LQR stabiliser is part of the primitive, not an external controller bank. Represent the active library item as a primitive-controller variant with `primitive_variant_id`, `primitive_id`, `entry_role`, `controller_id`, reference state/command, finite horizon, Q/R JSON, gain checksum, exit checks, metrics, failure labels, and claim status.

W0/W1 dense generation develops a rich primitive-controller variant library and tunes the primitive-local LQR inside every generated variant. Preserve all W0/W1 variants and evidence rows; do not reduce them to a 12--24 retained shortlist. W2 and W3 replay fixed variants to eliminate or downgrade failures without regenerating or retuning LQR controllers. Clustering, merging, or no-clustering library-size cases occur only after W3 as the required four-case cross-study. The mixed primitive-start distribution includes launch and in-flight starts; do not reject an in-flight-only primitive solely because it cannot start from the broad launch gate. Late validation uses frozen post-W3 variants and must not hide Q/R or gain retuning.


Current overwrite for active primitive timing:

```text
finite_horizon_s = 0.100
controller_input_slots_per_primitive = 5
controller_input_update_period_s = 0.020
```

Every primitive-controller variant generated in the next W0/W1 dense cycle must use the 0.10 s horizon and record the 5 controller-input slots at a 20 ms controller update-period contract. Any code path that silently keeps older long horizons must be deleted, rewritten, or marked retired_not_active before evidence generation.


## Core standard

Code must be:

- correct in frames, signs, and units;
- efficient enough for archive-scale simulation;
- rich in meaningful comments;
- traceable through manifests, metrics, and tests;
- narrow in scope.

## Rich-comment rule

Good comments explain why a line exists or what physical assumption it encodes.

Use comments for:

- units and coordinate frames;
- sign conventions;
- state and command order;
- context-feature definitions;
- controller timing assumptions, including W0/W1 feedback latency, command timing, and actuator lag;
- latency case meaning;
- safety gate meaning;
- belief update meaning;
- archive row classification;
- why a failure label is assigned;
- why a result is simulation-only or hardware-eligible.

Avoid comments that restate syntax.

## Naming and units

Use suffixes:

```text
_m, _s, _rad, _deg, _m_s, _rad_s, _kg, _N, _Nm
```

Use meaningful fields:

```text
environment_id
context_feature_vector
primitive_parameters
belief_before
belief_after
minimum_wall_margin_m
energy_residual_m
lift_dwell_time_s
termination_cause
claim_status
```

Avoid ambiguous names such as `data`, `val`, `result2`, `temp`, `out`, or `status_flag` unless the local context is trivial.

## Runtime and storage

Dense runs must use:

- chunked execution;
- resumable chunk manifests;
- compressed table partitions;
- CPU worker count chosen by profiling and memory guardrails;
- checksums and table manifests;
- no full-memory final rollout table.

Every generated file should stay below 100 MB. Prefer below 75 MB. If a table or figure may exceed 100 MB, split or simplify before writing.

## Environment context code

An environment-context encoder should expose local physical features instead of named environment branches.

Minimum useful outputs:

```text
w_cg_m_s
w_wing_mean_m_s
delta_w_lr_m_s
spanwise_w_gradient_m_s_per_m
w_local_uncertainty_m_s
lift_score
wall_margin_m
floor_margin_m
ceiling_margin_m
speed_margin_m_s
attitude_margin_rad
```

The encoder should not contain separate online logic for particular fan layouts.


## Directional 3D residual memory code

The active belief implementation must be 3D and directional-aware. It should store residual evidence relative to the current model and local flow context rather than an arena-specific fan map. Minimum fields:

```text
mean_lift_residual_m_s
mean_energy_residual_m
mean_dwell_residual_s
observation_count
uncertainty
last_update_launch_index
```

Keep the implementation simple and auditable: dataclasses, deterministic binning or compact sparse tables, explicit units, and clear update equations. The old 2D 4 x 4 memory may remain only as a labelled baseline or retired diagnostic.

## Safe exploration/exploitation code

The governor may add an exploration or uncertainty bonus only after a primitive passes entry, safety, and viability checks. Log the exploit score, memory residual bonus, exploration bonus, total score, rank without memory, rank with memory, and selected primitive. If memory or exploration never changes ranking, the result must be reported as not passed.

## Post-W3 library-size cross-study code

Post-W3 code must support four library-size cases:

```text
heavy_cluster
balanced_cluster
light_cluster
no_cluster
```

All four cases must consume the same W3-surviving evidence and must not mutate controller gains, Q/R weights, references, horizons, entry roles, controller IDs, or primitive variant IDs.

## Primitive outcome code

Outcome datasets should preserve W0/W1 rich evidence, W2/W3 survival evidence, and post-W3 library-size case labels. They should record:

```text
state
context features
primitive family
primitive parameters
accepted / weak / failed / rejected / blocked
boundary_terminal for retained x/y boundary exits
continuation_status
episode_terminal_status
episode_utility_label
energy residual
dwell time
minimum margin
termination cause
failure label
```

Do not promote diagnostic, open-loop, schema-only, command-template, or blocked rows into mission evidence. Keep continuation-valid labels separate from episode-terminal-useful labels. X/y boundary exits may be useful terminal episode evidence, but they are not downstream-continuation successes. Do not run clustering or small-library selection on W0/W1 outputs; compression is a post-W3 operation.


The next active dense generation must archive rejected old results first, label v4.10-style outputs as diagnostic_not_passed, then rerun W0/W1 with the 0.10 s horizon setting. New dense evidence must not overwrite the v4.10-style diagnostic outputs.

## Validation and tests

Every evidence-generating script should have at least one of:

- smoke test;
- schema test;
- row-count check;
- manifest consistency check;
- file-size audit;
- seed reproducibility check;
- physics/sign/unit audit.

A run is not evidence unless it produces traceable outputs.

## Final response standard for coding tasks

Report:

1. files changed;
2. interfaces added or changed;
3. tests/checks run;
4. evidence files generated;
5. claim boundaries preserved;
6. remaining risks.
