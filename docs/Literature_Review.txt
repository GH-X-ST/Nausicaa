## Current controlling update - transition-aware primitive method

The active thesis algorithm is now deliberately simple and transition-based:
`R5 -> R7 -> R8 -> R10 -> R11 -> Reality`.
R9 remains internal preflight only and is not thesis-facing evidence, not a final claim gate, and not required before R10.

Core method:
1. R5 learns transition-aware primitive variants under rich randomized synthesis.
2. R7 validates transition compatibility under held-out randomisation with frozen LQR controllers.
3. R8 compresses only transition-compatible primitives using coverage-aware medoid selection.
4. R10 tunes the viability governor with residual updraft adaptation on the frozen R8 library.
5. R11 is strict held-out validation of the frozen R10 governor.
6. Reality is a later real flight transfer attempt only after simulation gates pass.

Every primitive is treated as a transition object with primitive_id, entry_class, exit_class, transition_success_probability, hard_failure_probability, updraft_gain_proxy, flight_time, and controller metadata. The governor asks: given the current state class, which primitive has a valid entry class and predicts a viable exit class?

Use these compact state classes only unless later data proves more are needed: launch_gate, post_launch_degraded, inflight_stable, boundary_near, recoverable_degraded, safe_terminal, and hard_failure. boundary_near is a route state, not automatic failure. hard_failure is the failure class.

Primitive-role transition contract:
- launch_capable: entry launch_gate; required exit post_launch_degraded or inflight_stable.
- inflight_only: entry inflight_stable; required exit inflight_stable, boundary_near, or safe_terminal.
- terminal_or_recovery: entry boundary_near or recoverable_degraded; required exit inflight_stable or safe_terminal.

No primitive may pass R7 solely on local rollout success. R7 must reject or downgrade locally accepted rows that are not chain-compatible for the declared primitive role. R5 reports transition coverage but does not claim primitive validity from aggregate local accepted rows.

R8 selection objective is the smallest existing validated set that covers useful entry_class -> exit_class transitions with low hard_failure_probability. Do not average Q/R, synthesize new controllers, mix launch/inflight/recovery roles, or cluster before R7.

The active governor path is transition_viability_governor: classify current state, filter matching primitive entry class, reject high hard_failure risk, score transition viability plus updraft gain plus flight time plus residual-memory correction, execute the best primitive, then update residual memory. Residual memory is only a small modifier: predicted_updraft_gain = library_prediction + residual_memory_correction. It must not change state classification or entry/exit rules.

Thesis-core comparisons are reduced to no memory versus residual memory histories h5, h20, and h100, with at most one safe-explore ablation. The old 14 policy/history matrix, large repeated history-launch matrices, secondary launch score, exhaustive candidate logs, multiple diagnostic archives, and R9 are audit/ablation/appendix material only.

Memory is reinitialised per final test row. Final scoring is computed only from the final held-out rollout path. full_safe_success is the strict claim gate. updraft_gain_weight is the active soft reward; expected_energy_residual_m is a legacy net-energy audit alias. Low speed is not an active governor rejection reason. Energy loss is also not a hard-failure reason. sample_count remains an outcome-model coverage audit field. R11 consumes the frozen governor config written by R10.

Preserve these constraints: LQR-only primitive controllers, 0.100 s horizon, 5 controller-input slots, 20 ms controller update, no PD/PID, no TVLQR active workflow, no new controller family, no fan-layout-specific controller logic, no hidden speed gate, no energy-loss hard failure, no stale v4.10 evidence, and no hardware readiness, real-flight transfer, mission success, full autonomy, or memory-improvement claim until R11 and later real flight evidence pass.

## Purpose

Use this guide for related-work searching, literature-review rewriting, and reference selection. The goal is to support a robotics-style thesis or paper about environment-conditioned primitive selection for repeated-launch fixed-wing updraft exploitation.

## Search priorities

Use this order:

1. Existing thesis bibliography and uploaded PDFs.
2. High-quality robotics, UAV, and control papers with real experiments.
3. Papers on fixed-wing soaring, local lift exploitation, gust-aware flight, and wind-field-aware control.
4. Papers on viability, safety filters, motion primitives, primitive libraries, and reference governors.
5. Papers on sim-to-real transfer, domain randomisation, and environment generalisation.
6. Papers on post-W3 archive compression, clustering, merging, and representative rollout selection.
7. Preprints only when no peer-reviewed source exists or the topic is very recent.

## Required literature groups

### 1. Sim-to-real transfer under aerodynamic forcing

Use this group to justify domain randomisation as a survival-filtering and held-out validation tool: simulated variation can remove or downgrade weak primitive-controller variants, but final validation must freeze the post-W3 selected library-size condition or explicitly declare adaptation.

Purpose: justify why measured updrafts, latency, and wing-scale wind descriptors matter.

Look for:

- fixed-wing sim-to-real control;
- aerodynamic model mismatch;
- wind-field-aware control;
- paired simulation and real-flight replay.

### 2. Local lift exploitation

Purpose: connect the indoor updraft to real-life flow near terrain, buildings, slopes, or obstacles without claiming exact outdoor equivalence.

Use wording:

```text
local vertical-flow exploitation near terrain-induced or built-environment flow features
```

Avoid:

```text
the fan updraft is an urban thermal
```

### 3. Autonomous soaring and wind-field inference

Purpose: show that lift exploitation and partial lift knowledge are real aerospace robotics problems.

Use as precedent, not as the final objective. The arena is too small for continuous thermalling or ridge soaring.

### 4. Viability and safety filtering

Purpose: justify the governor as a constraint filter over candidate primitives.

Explain:

- admissible sets;
- safe candidate rejection;
- separation between candidate generation and safety filtering;
- why logging rejections matters.

### 5. Manoeuvre primitives and libraries

The feedback stabiliser should be treated as embedded inside the primitive. Use this group to support primitive-controller variants rather than a free-standing controller bank.

Purpose: justify short actions with entry sets, feedback, exit checks, and metrics.

Use this group to distinguish the project from generic waypoint following.

### 6. Environment-conditioned decision making

Purpose: support the new method abstraction.

Look for:

- context-conditioned policies;
- local feature-conditioned action selection;
- environment-adaptive primitive or behaviour libraries;
- real robot tests across changed environment parameters.

### 7. Domain randomisation and generalisation

Purpose: support W3 as fixed-LQR environment-randomised survival filtering and later held-out full-loop validation, including fan position, fan power, fan number, updraft amplitude, launch-state variation, glider-plant variation, latency jitter, command timing, and actuator delay.

### 8. Repeated experiments and episodic learning

Purpose: support repeated-launch learning from physical rollouts.

Look for:

- repeated physical trials;
- belief update or confidence decay;
- non-identical initial conditions;
- real-world data improving future decisions.

### 9. Perching and LQR-Trees, as contrast only

Perching is not the final objective. Use it to explain what the project does not do:

- no terminal landing target;
- no post-stall terminal capture requirement;
- no attempt to reproduce LQR-Tree perching.

## Paragraph structure

Use:

```text
Prior work does X under assumptions Y. This supports our use of Z. However, those works do not address A, which motivates the present method.
```

Example:

```text
Autonomous soaring studies show that useful lift can be inferred and exploited by fixed-wing vehicles, but they usually assume larger outdoor volumes or longer flight times. This thesis instead studies whether a small glider can select 0.100 s primitives from local flow context inside a confined measured updraft.
```

## Avoid generic literature writing

Do not write:

```text
Recent years have witnessed increasing attention...
Many researchers have proposed many methods.
```

Prefer specific comparisons.

## Reference quality check

For every candidate reference, record:

```text
reference type: foundational / method / experimental / review / contrast
system: real robot / simulation / wind tunnel / theory
why it matters
what it does not support
replacement needed: yes/no
```

Discard references that cannot support a claim or contrast.
