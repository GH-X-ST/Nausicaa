## Active Transition-Aware Thesis Workflow

The active thesis workflow is `R5 -> R7 -> R8 -> R10 -> R11 -> Reality`. R9 remains internal preflight only and is not thesis-facing evidence. R10 tunes the viability governor with residual updraft adaptation, and R11 is the held-out validation gate.

Launch is an entry regime, not a separate controller family. The active primitive catalogue has exactly eight manoeuvre families: `glide`, `recovery`, `lift_entry`, `lift_dwell_arc`, `mild_turn_left`, `mild_turn_right`, `energy_retaining_bank`, and `safe_exit_or_recovery_handoff`. Retired `launch_capture_*` IDs are archive aliases only and must not appear in active evidence.

Every primitive is treated as a transition object. R5 is robust transition-aware primitive / transition-object Q/R plus primitive-reference-bias training across five entry start families with exact dense proportions per primitive/candidate/environment: 40 `launch_gate`, 25 `inflight_nominal`, 15 `inflight_lift_region`, 10 `inflight_boundary_near`, and 10 `inflight_recovery_edge`. The dense evaluation target remains `8 * 32 * 3 * 100 = 76,800` rows, but row count alone is not a pass condition. Candidate 0 is nominal, candidates 1-7 are named physical anchors with small interpretable reference biases, and candidates 8-31 are deterministic Latin-hypercube log multipliers over the seven grouped LQR weights plus bounded pitch, bank, and speed reference biases. R5 writes `r5_transition_candidate_training_summary.csv`, `r5_transition_selected_for_r7.csv`, `r5_transition_pareto_front.csv`, and `r5_transition_training_manifest.json`, then freezes only selected transition objects for R7 while keeping the full candidate bundle as audit evidence.

R7 is held-out transition validation of the frozen R5-selected transition objects. No primitive may pass R7 solely on local rollout success; no primitive may pass R5 or R7 from dense row count or aggregate primitive success across entry classes. A controller can survive for one `primitive_id + entry_class` and fail for another. R8 compresses transition objects grouped by `primitive_id` and `transition_entry_class` using coverage-aware medoid selection without averaging Q/R, K, references, or controller IDs.

The governor classifies the current state, filters representatives by validated `transition_entry_class`, rejects high hard-failure risk, scores transition probability plus updraft gain plus flight time plus residual-memory correction, executes the best transition object, and updates case-local residual memory. Step 0 has `current_state_class = launch_gate`, so it selects only transition objects validated for `entry_class = launch_gate`; there is no launch-specific primitive family route.

Residual memory is a small case-local modifier: `predicted_updraft_gain = library_prediction + residual_memory_correction`. It must not override state classification or entry/exit compatibility. Memory is reinitialised per final test row. Final scoring is computed only from the final held-out rollout path. There is no hidden speed gate, no energy-loss hard failure, no PD/PID, no TVLQR, and no fan-layout-specific controller logic.

Core comparison is no memory versus residual-memory histories such as h5, h20, and h100, with safe-explore only as optional ablation. Hardware readiness, real-flight transfer, mission success, autonomy, and memory-improvement claims require R11 and later real-flight evidence.

## Purpose

Use this guide for MATLAB scripts, Arduino/receiver code, latency tests, command-path tests, and hardware-facing tools.

Hardware-facing code should be simple, traceable, and conservative. It should prioritise logging, timing clarity, and repeatability over cleverness.

## Current method relevance

Hardware tools should support environment-conditioned primitive selection by recording:

```text
measured initial state
episode ID
primitive command or selected primitive
command timing
surface response timing
termination cause
local environment identifier
real-flight replay metadata
```

W labels are fidelity/evidence layers, not online controller branches. W0 is dry air only. Active pass-gated R5 W1 evidence now uses annular-GP randomized single-fan and four-fan training blocks with the same randomisation dimensions as W3 held-out validation; axisymmetric Gaussian plume is diagnostic-only. W2 is optional GP-corrected annular-Gaussian diagnostic replay, and W3 is randomised GP-corrected annular-Gaussian held-out validation. Fan count, fan shift, and fan power are environment instances. W0/W1 evidence should already use the feedback-latency, command-timing, and actuator-lag assumptions that affect primitive-local LQR tuning.

## Comments

Use rich comments for:

- timing assumptions;
- serial/PPM/PWM signal meaning;
- units;
- channel order;
- calibration constants;
- safety or abort conditions;
- measurement trigger definitions.

Do not remove useful comments. Hardware code often needs more comments than simulation code because failure modes are less visible.

## Naming

Use explicit channel names:

```text
aileron_cmd
elevator_cmd
rudder_cmd
surface_response_time_ms
command_onset_time_ms
```

Avoid ambiguous names such as `ch1`, `val`, or `out` unless immediately mapped in a comment.

## Timing tests

Every timing test should record:

```text
command timestamp
observed response timestamp
sample rate
filtering or detection rule
median latency
P95 latency
max latency
hardware route
```

## Hardware experiment rule

Do not change control code and hardware code simultaneously unless necessary. If both change, report which result can still be compared with earlier evidence.

## Launch and repeated-episode support

Hardware code should support:

- Vicon-valid start trigger;
- launch-state logging;
- command timing logs;
- abort command path;
- episode ID tagging;
- selected primitive ID tagging;
- selected primitive_variant_id / controller_id tagging;
- real-flight replay matching.

Do not add complex onboard autonomy unless explicitly requested.
X/y boundary exits should be logged as terminal episode evidence when they occur. They are not continuation-valid states, and they must not be deleted merely because the episode terminates at a lateral boundary.

## File-size rule

Hardware logs should be split before any file exceeds 100 MB. Prefer compact CSV summaries plus compressed raw logs when high-rate data are needed.
