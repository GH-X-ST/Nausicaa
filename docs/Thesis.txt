## Purpose

Use this guide when drafting or revising thesis text, captions, abstracts, and conclusions. The target style is a robotics-journal-style engineering thesis: real system, measured environment, experimental protocol, baselines, failure cases, and reproducibility.

## Current project framing

Use this mission framing:

> Environment-conditioned primitive selection for repeated-launch lift capture and finite-horizon dwell in a measured uncertain indoor updraft.

The final evaluation treats each physical launch as an episode. The glider starts from a fixed nominal launch gate, the actual initial state is measured, local flow-context features are computed, the viability governor selects among post-W3 validated short primitives, and the flight log updates a directional 3D residual lift belief for later launches.

Method wording should treat the LQR stabiliser as part of each primitive. The active library item is a primitive-controller variant: entry set, reference, finite horizon, local time-invariant LQR Q/R/K, exit checks, metrics, and failure labels. W0/W1 dense generation tunes each primitive-local LQR and preserves a rich library; W2 and W3 replay fixed variants to remove cases that fail under higher-fidelity and domain-randomised conditions; post-W3 library-size cross-study produce the compact robust library used by the governor. Late validation freezes the selected library-size condition, governor, selector, and memory logic before full-loop randomised simulation and real repeated launches. Do not describe W0/W1 as an external controller-bank step or move clustering before W2/W3.

## What the thesis is not

Do not frame the thesis as:

```text
perching
generic thermal soaring
generic path following
all-arena exploration
fan-layout-specific controller design
same-flight recapture
long primitive-chain construction
```

## Writing style

Use compact, precise, traceable engineering prose.

Each results paragraph should follow:

```text
claim -> metric -> baseline comparison -> limitation
```

Avoid vague claims such as:

```text
significantly improves
robust performance
highly effective
```

unless a metric and test condition directly support them.

## Claim labels

Use:

```text
simulation_only
hardware_shakedown
real_flight_evidence
partial_transfer
negative_transfer
instrumentation_limited
not_tested
```

Do not claim real-flight transfer, mission success, hardware readiness, or full robustness unless direct evidence supports it.

## Figure captions

Good caption pattern:

```text
Environment-conditioned primitive outcome map under measured-updraft replay. Accepted cases occupy high wing-mean wind and adequate wall-margin regions, while low-speed cases remain rejected by the governor.
```

Bad caption pattern:

```text
Results of the controller.
```

## Reviewer self-check

Before finalising a paragraph, ask:

1. What is the precise claim?
2. What measurement, simulation, or reference supports it?
3. What is the baseline?
4. What limitation remains?
5. Could this sentence appear in any generic UAV paper? If yes, make it project-specific.
