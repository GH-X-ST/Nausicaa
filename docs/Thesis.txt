## Active Transition-Aware Thesis Workflow

The active thesis workflow is `R5 -> R7 -> R8 -> R10 -> R11 -> Reality`. R9 remains internal preflight only and is not thesis-facing evidence. R10 tunes the viability governor with residual updraft adaptation, and R11 is the held-out validation gate.

Launch is an entry regime, not a separate controller family. The active primitive catalogue has exactly eight manoeuvre families: `glide`, `recovery`, `lift_entry`, `lift_dwell_arc`, `mild_turn_left`, `mild_turn_right`, `energy_retaining_bank`, and `safe_exit_or_recovery_handoff`. Retired `launch_capture_*` IDs are archive aliases only and must not appear in active evidence.

Every primitive is treated as a transition object. R5 is robust transition-aware primitive / transition-object Q/R plus primitive attitude/bank reference-bias training across five entry start families with exact dense proportions per primitive/candidate/environment: 40 `launch_gate`, 25 `inflight_nominal`, 15 `inflight_lift_region`, 10 `inflight_boundary_near`, and 10 `inflight_recovery_edge`. The dense evaluation target remains `8 * 32 * 3 * 100 = 76,800` rows, but row count alone is not a pass condition. Candidate 0 is nominal, candidates 1-7 are named physical anchors with small interpretable attitude/bank reference biases, and candidates 8-31 are deterministic Latin-hypercube log multipliers over the seven grouped LQR weights plus bounded pitch and bank reference biases. Speed-bin selection is for local model scheduling only; active primitives must not chase speed as a hard reference. R5 writes `r5_transition_candidate_training_summary.csv`, `r5_transition_selected_for_r7.csv`, `r5_transition_pareto_front.csv`, and `r5_transition_training_manifest.json`, then freezes only selected transition objects for R7 while keeping the full candidate bundle as audit evidence.

R7 is held-out transition validation of the frozen R5-selected transition objects. No primitive may pass R7 solely on local rollout success; no primitive may pass R5 or R7 from dense row count or aggregate primitive success across entry classes. A controller can survive for one `primitive_id + entry_class` and fail for another. R8 compresses transition objects grouped by `primitive_id` and `transition_entry_class` using coverage-aware medoid selection without averaging Q/R, K, references, or controller IDs. R8 must also preserve distinct W3-surviving local LQR speed-bin coverage within each primitive/entry-class group up to the case representative budget; speed-bin collapse is a library-coverage failure, not an LQR-principle failure.

The governor classifies the current state, filters representatives by validated `transition_entry_class`, rejects high hard-failure risk, scores transition probability plus updraft gain plus flight time plus residual-memory correction, executes the best transition object, and updates case-local residual memory. Step 0 has `current_state_class = launch_gate`, so it selects only transition objects validated for `entry_class = launch_gate`; there is no launch-specific primitive family route.

The inner LQR remains a stabilising tracker around the primitive-defined local reference. It must not become the manoeuvre planner: primitive transition objects define manoeuvre intent and R5/R7/R8 decide which transition objects are valid, while the governor selects among those frozen transition objects. Same-start trajectory plots are diagnostic-only sanity checks for controller behaviour and do not replace R5/R7 transition evidence.

Residual memory is a small case-local modifier: `predicted_updraft_gain = library_prediction + residual_memory_correction`. It must not override state classification or entry/exit compatibility. Memory is reinitialised per final test row. Final scoring is computed only from the final held-out rollout path. There is no hidden speed gate, no energy-loss hard failure, no PD/PID, no TVLQR, and no fan-layout-specific controller logic.

Core comparison is no memory versus residual-memory histories such as h5, h20, and h100, with safe-explore only as optional ablation. Hardware readiness, real-flight transfer, mission success, autonomy, and memory-improvement claims require R11 and later real-flight evidence.

## Purpose

Use this guide when drafting or revising thesis text, captions, abstracts, and conclusions. The target style is a robotics-journal-style engineering thesis: real system, measured environment, experimental protocol, baselines, failure cases, and reproducibility.

## Current project framing

Use this mission framing:

> Environment-conditioned primitive selection for repeated-launch lift capture and finite-horizon dwell in a measured uncertain indoor updraft.

The final evaluation treats each physical launch as an episode. The glider starts from a fixed nominal launch gate, the actual initial state is measured, local flow-context features are computed, the viability governor selects among post-W3 validated short primitives, and the flight log updates a directional 3D residual lift belief for later launches.

Method wording should treat the LQR stabiliser as part of each primitive. The active library item is a primitive-controller variant: entry set, reference, finite horizon, local time-invariant LQR Q/R/K, exit checks, metrics, and failure labels. W0/W1 dense generation tunes each primitive-local LQR and preserves a rich library; optional W2 diagnostic replay and R7/W3 held-out replay remove or downgrade cases that fail under higher-fidelity and domain-randomised conditions; post-W3 library-size cross-study produces the compact robust library used by the governor. Late validation freezes the selected library-size condition, governor, selector, and memory logic before full-loop randomised simulation and real repeated launches. Do not describe W0/W1 as an external controller-bank step or move clustering before R7/W3.

Do not write that the LQR tuning is globally optimal. The active claim is
narrower: R5 evaluates a deterministic Q/R plus attitude/bank reference-bias candidate surface
and selects empirically robust transition objects, then R7 tests those fixed
objects under held-out randomisation.

## What the thesis is not

Do not frame the thesis as:

```text
perching
generic thermal soaring
generic path following
all-arena exploration
fan-layout-specific controller design
same-flight recapture
long primitive-chain construction
```

## Chapter-level implications

### Chapter 1

Update the objective and research questions around:

```text

local flow-context features

viability filtering
sim-to-real transfer under updraft uncertainty
```

### Chapter 2

Use related work to support:


```text

```

## Writing style

Use compact, precise, traceable engineering prose.

Each results paragraph should follow:

```text
claim -> metric -> baseline comparison -> limitation
```

Avoid vague claims such as:

```text
significantly improves
robust performance
highly effective
```

unless a metric and test condition directly support them.

## Claim labels

Use:

```text
simulation_only
hardware_shakedown
real_flight_evidence
partial_transfer
negative_transfer
instrumentation_limited
not_tested
```

Do not claim real-flight transfer, mission success, hardware readiness, or full robustness unless direct evidence supports it.

## Figure captions

Good caption pattern:

```text
Environment-conditioned primitive outcome map under measured-updraft replay. Accepted cases occupy high wing-mean wind and physically valid boundary regions, while rejected cases are labelled by governor reason rather than by speed margin.
```

Bad caption pattern:

```text
Results of the controller.
```

## Reviewer self-check

Before finalising a paragraph, ask:

1. What is the precise claim?
2. What measurement, simulation, or reference supports it?
3. What is the baseline?
4. What limitation remains?
5. Could this sentence appear in any generic UAV paper? If yes, make it project-specific.

### R9 launch-gate repair wording

When discussing the current blocked simulation evidence, state that R9 revealed a transition-compatibility limitation: locally surviving primitives were not being judged strongly enough by their entry-class -> exit-class handoff. The repair is not a separate launch-capture controller family. Launch is now a `launch_gate` entry regime for the same eight manoeuvre families, and R7 must reject or downgrade any primitive-entry transition that cannot hand off to a compatible exit class. This should be presented as a project-aligned transition-library repair, not as LQR-tree, funnel-library, TVLQR, or a new controller family.

When describing the plant model, state that the active simulation uses a simplified as-built rigid-body strip-aerodynamics model with generated deterministic mass/inertia constants, semi-empirical strip aerodynamics, local-speed scheduled LQR linearisation, first-order actuator states, and measured-style Vicon/command latency constants. Do not imply CFD, aeroelastic modelling, or flight-identified aerodynamic correction tables unless those are later added and audited.
